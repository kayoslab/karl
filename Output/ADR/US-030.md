  "id": "ADR-007",
  "title": "Multi-Instance Isolation and Concurrency Contract",
  "status": "Accepted",
  "context": "ADR-001 documents a single LOCK file at the repository root for single-instance protection. US-030 introduces multi-instance mode where N concurrent karl processes share the same workspace. Without a documented contract for LOCK file naming, LOCK file content format, ticket collision avoidance, and concurrent writes to shared artifacts (prd.json, progress.md), concurrent instances would corrupt each other's state.",
  "decision": "When multi-instance mode is active (--instances N, N > 1), the following rules apply:\n\n**LOCK file naming**: Each instance uses `LOCK.<INSTANCE_ID>` (e.g. `LOCK.1`, `LOCK.2`). When INSTANCE_ID is 1 or unset (single-instance default), the file is named `LOCK` to preserve backward compatibility.\n\n**LOCK file content format**: All LOCK files (single and multi-instance) use structured content:\n```\nPID=<pid>\nINSTANCE=<id>\nTICKET=<ticket_id>\n```\nThe TICKET field is written at instance startup (empty) and updated atomically via `lock_update_ticket()` when a ticket is claimed.\n\n**Ticket collision avoidance**: Before starting work on a ticket, each instance reads all sibling `LOCK.*` files and checks their TICKET field. If another live instance (PID alive) has claimed the ticket, it is skipped and the next eligible ticket is selected. Stale LOCK files (dead PID) are ignored.\n\n**Atomic ticket claiming**: To close the race window between `prd_select_next()` and `lock_update_ticket()`, a per-ticket claim sentinel `Output/<ticket_id>/.claimed` is created with `set -C` (noclobber). The first instance to create this file wins the ticket; others detect the file and skip.\n\n**Concurrent writes to prd.json**: A `prd.lock` file at the workspace root is acquired with noclobber before any prd.json write and released immediately after. This serialises concurrent prd.json updates.\n\n**Concurrent writes to progress.md**: Appends are guarded with `flock` when available (Linux); macOS fallback is unguarded append. Since each instance writes a complete line atomically in most shells, line interleaving risk is low but documented.\n\n**Cleanup**: `--clean` must glob `LOCK*` (not just `LOCK`) to remove all instance lock files. The `.claimed` sentinels under `Output/` are removed when their ticket's output directory is cleaned.\n\n**Single-instance default**: When `--instances` is absent or `--instances 1`, behaviour is identical to the pre-US-030 contract. No LOCK.* files, no collision detection, no prd.lock.",
  "consequences": "- ADR-001 LOCK entry is superseded for multi-instance mode; single-instance behaviour is unchanged.\n- The structured LOCK file format is a breaking change to any external tooling that reads the LOCK file as a raw PID.\n- The `.claimed` sentinel pattern introduces a new ephemeral file type under `Output/<ticket_id>/` not previously documented in ADR-001; workspace bootstrap and clean must account for it.\n- `flock` availability is platform-dependent; progress.md append safety is best-effort on macOS without coreutils.\n- All multi-instance behaviour must be covered by `tests/lock.bats` and `tests/loop.bats`; any change to LOCK naming, content format, or collision semantics requires updating this ADR and those tests."
