# ADR-006: Logging Contract

## Status
Accepted

## Context
All lib modules emit echo statements for progress, diagnostics, and errors. Without a centralized logging contract, there is no way to control output verbosity at runtime: every internal diagnostic line reaches the terminal regardless of user preference, making karl noisy during normal use and making it impossible to silence internal output without modifying source files.

## Decision
All terminal output from karl **must** go through one of two functions defined in `lib/log.sh`:

- `karl_log <msg>` — always emits to stdout; used for high-level phase banners, completion events, and errors.
- `karl_log_verbose <ticket_id> <msg>` — emits to stdout **only** when `KARL_VERBOSE=true`, prefixed with an ISO-8601 timestamp and the ticket identifier (e.g. `2026-03-25T14:00:00Z [US-027] msg`).

Direct `echo` calls for diagnostic or internal-state output are prohibited in lib modules; they must be replaced with the appropriate logging function.

Error output destined for stderr (`>&2`) is exempt and may remain as direct `echo`/`printf` calls.

### CLI surface
`karl.sh` accepts `--verbose` as a command-line flag. When present, the entrypoint sets and exports `KARL_VERBOSE=true` before sourcing any lib. The default is `KARL_VERBOSE=false`.

### Environment variable contract
| Variable | Values | Set by |
|---|---|---|
| `KARL_VERBOSE` | `true` / `false` (default) | `karl.sh` entrypoint via `--verbose` flag |

`KARL_VERBOSE` is exported by the entrypoint so all sourced lib scripts inherit it. Lib scripts must not set or override `KARL_VERBOSE`.

### Source directive requirement
Every lib file that calls `karl_log` or `karl_log_verbose` must include a shellcheck source directive for `lib/log.sh` to satisfy SC2154.

## Consequences
- Any new lib module that emits diagnostic output must use `karl_log_verbose`; any new lib module that emits a high-level event must use `karl_log`. Direct `echo` for diagnostics is a contract violation.
- When lib files are sourced directly in tests (not via `karl.sh`), `KARL_VERBOSE` will be unset, which is treated as `false` — verbose output is suppressed by default in the test environment. Tests that assert verbose output must explicitly set `KARL_VERBOSE=true` in setup.
- Existing tests that assert on specific output strings must be audited to ensure no asserted message has been demoted to `karl_log_verbose`; suppressed messages in tests will cause false-negative assertion failures if not caught.
- All logging behaviour is covered by `tests/log.bats`; any change to the log format, the environment variable name, or the CLI flag requires updating this ADR and the tests.
